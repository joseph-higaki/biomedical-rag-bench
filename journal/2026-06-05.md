## Decisions (and why)
- **Constrained candidate-query sampling** over sample-then-filter — pool valid by
  construction, rejection-free (critical for paired types).
- **Fan bound in the candidate query** (HAVING) — reproducible form of step-2's
  hand-picked bounded seeds (CNS expresses ~11k genes otherwise).
- **Anchor-fan bound** for paired — intersection/difference ≤ |A| kills the
  hub-anchor query explosion.
- **`target_type` to mirror the `.rq`'s type filter** — `participates` is polymorphic
  (Pathway is the minority vs BiologicalProcess); without it overlap ≠ intersection →
  correctness bug + ~50× slowdown. **Session's central lesson: a candidate query must
  mirror every constraint the `.rq` applies.**
- **Multi-hop answer post-check** (03/04) — the head-edge fan bounds the wrong hop.
- **Balanced true/false boolean** (09) — the ASK `.rq` is the source of truth; the
  bridge query only steers which label to look for.
- **`validate.py` in `eval/produce/`** (production concern); pure `validate_records`
  + lazy `yaml` import so the hermetic tests need no `produce` extra.
- **Fuzzy questions hand-authored**, graph-anchored label-lookup `.rq`, balanced node
  types (2 gene / 2 compound / 2 disease).
