## Decisions (and why)
- **Temperature is a logged factor, orthogonal to the judge.** "Deterministic-judged" describes the scorer, not the generator; conflating them is what hid the drift. Logged separately so each row is self-describing.
- **Pin temp 0 (reproducible-by-default), env-overridable.** Measures modal behavior — accepted trade-off (can lock in generator hedging) for a stable benchmark; sampling stays available as a factor.
- **Writer needs its own pin** because graph_sparqlgen chains two LLM calls; a hot writer samples a different query → different result set → different context.
- **`ORDER BY` fix only — don't raise the limit knobs, don't add priority heuristics.** Bigger caps = bigger haystack + more cost (data already shows more 2hop context hurts); priority heuristics would make the *naive* baseline clever and blur the naive-vs-query-execution contrast. Keep 2hop honestly naive; let it settle at its true (hollow) 13.
- **The audit was worth exactly one repeat** — the 2nd run found the bug; a 3rd would only re-roll the dice. Nondeterminism anywhere upstream of the judge makes a score a draw.
