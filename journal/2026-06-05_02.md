## Decisions (and why)
- **`.env` via `find_dotenv`, not `parents[2]` or a config module.** Single call, no new dependency, depth-agnostic. A shared `config.py` (immutable `settings`, fail-fast, `.env` as dev fallback) is the right pattern but premature with one consumer — deferred to step 5 and noted in the build order.
- **Single-source the distribution on YAML (generate the table), not keep an independent target + ±2 drift guard.** User's call: collapsing target/actual into one derived number avoids maintaining a 4th doc dimension (on top of type / hypothesis / scoring). Reuses the existing generate-docs-from-YAML pattern.
- **Producer trace as a generated artifact, organized by `type_id`.** A hand-written walkthrough would drift; `--explain` reuses the real path so it can't. Coverage is one-per-type (not one-per-code-path), keyed by the existing taxonomy spine — deliberately *not* introducing "sampling regime" as a new visible grouping.
- **Flowchart-styled-as-C4, not official `C4Container`.** GitHub renders Mermaid C4 but its layout is experimental and macro support is incomplete (verified by search); a styled flowchart renders reliably with full layout control.
