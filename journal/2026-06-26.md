## Decisions (and why)
- **Hash + human version label, not hash-only** (user pick). `version` is orderable so analytics can report "prompt improvement" / A/B (T1); `sha256` is content-addressed identity; a version held constant across a changed sha is therefore a detectable silent edit.
- **Defer the abstention flag to Theme B** (user pick). Refusal semantics (CWA on empty GT, generator-trust-context) belong with the refusal-floor investigation, not this exact-telemetry MINOR. Keeps the MINOR fully deterministic; cleaner scope boundary.
- **Per-strategy judge versioning, not per-family.** A fix to `set_match` (the T6 over-claim bug) bumps only `set_match-v2`; `string_match`/etc. stay honestly v1. A `deterministic-v2` would falsely re-version unchanged results — the comparability lie this work exists to prevent.
- **Complete prompt registry every run.** A closed_book run still records the writer/judge prompts of its build; the rows (`retriever`, `judge_id`) say which roles actually fired. The manifest is the prompt *snapshot*, consistent with how it already records one corpus/judge-roster.
- **Text in manifest, SHA per row.** Prompt text once per run (no per-JSONL-row bloat); rows carry only the SHA join key. Generator gets no per-row SHA (one per run; joins by `run_id`).
- **`prompt_record` lives in `retrievers/base.py`** — the dependency-free shared seam the harness already imports, so the judge and retriever both reach it without an inverted import or a new utils package.
- **`run_eval` (composition root) assembles the registry, not the harness** — preserves the harness's "injected with already-built roles" agnosticism.
- **Committed to `main`** (project convention); troubleshoot move parked separately per user.
