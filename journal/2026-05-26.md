## Decisions (and why)

- **Load the full graph via the REST statements endpoint, not the Workbench.**
  467 MB > the 200 MB Workbench upload cap; curl streams with no cap and is
  scriptable. `DELETE` + `POST` (add) over a single ambiguous `PUT` (replace) —
  two unambiguous steps matching the README's clear-then-load model.
- **Gitignore the full `hetionet.ttl`.** 467 MB, regenerable via `make
  ingest-rdf`; same class as `data/` and the smoke ttl. Committing would bloat
  history irreversibly. The repo carries the *generator*, not the artifact.
- **`pubmed_fetch.py` interface = `--entities <ttl>`, with `--limit` as an
  orthogonal cap.** Driving the fetch off the Turtle's entities (not a bare
  count) keeps the smoke *vector* set aligned with the smoke *graph* set — same
  100-edge slice → a combined graph+vector query can be tested on one coherent
  universe. It also exercises all 9 entity kinds, surfacing per-kind query
  problems early.
- **Smoke validates plumbing + query formulation, not retrieval quality.** 5 or
  191 docs is too few to judge retrieval; the goal at this rung is "the pipe is
  connected and the queries aren't garbage."
