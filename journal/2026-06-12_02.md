## Decisions (and why)
- **GraphDB recovery is `down`/`up`, not data restore** — the corpus was never lost; bind mounts diverged on the WSL restart. Flagged as a **recurring hazard** worth a readiness gate before runs (not built).
- **Re-run by splice, not `--types`** — re-running whole types would re-score already-good questions and risk fresh crashes; the surgical splice touches only the 3 errored rows.
- **Sonnet-writer is a negative result, kept as such** — refutes "a stronger writer raises retrieval quality"; once a writer emits valid SPARQL, capability stops mattering. Don't chase it; the real fix is case-insensitive label anchoring (deferred).
- **`writer_model_family` mirrors `generator_model_family`** — using raw `writer_model` would split a writer across its alias/resolved forms (the exact bug the `_family` normalization exists to prevent). Symmetric nomenclature is the correct fix.
- **Keep FINDINGS for interpretation only; evict the numbers.** Numbers belong to the data layer (results + notebook); a prose file is the wrong home for a value that drifts every temp-0 re-run. LATEST_RUN cannot replace FINDINGS (one run, auto-clobbered, no narrative).
- **Pin Step-2 figures to a canonical surface (haiku gen + haiku writer)** — the grain change introduced a writer-averaging trap for any retriever-only aggregate on sparqlgen; pinning one surface closes it.
- **Notebook `KeyError` was a stale kernel**, not a code bug — `from … import` reuses the cached module; restart-kernel-and-run-all is the fix.
