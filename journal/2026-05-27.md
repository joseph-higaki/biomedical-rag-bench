## Decisions (and why)

- **Split `ingest/` by representation, not by implementation.** `rdf/` and
  `vector/` mirror the project's core comparison; a future `lpg/` for Project 3's
  Neo4j path slots in naturally. Per-side documentation was outgrowing flat
  `ingest/README.md` and would have continued to. The split also makes the
  graph-side ops detail (license + REST load) live next to the code that needs
  it, not above the vector code that doesn't.
- **`pubmed_fetch.py` reads entities from the Turtle, not a live GraphDB.**
  Removes a runtime dependency: the vector side only needs `make ingest-rdf`
  to have produced the `.ttl`. Streams the 467 MB file line-by-line; the
  regex matches the serializer's exact two-line node form (`<term> a hetio:Kind ;`
  followed by `rdfs:label "…"`), so edge triples and RDF-star annotations
  don't match. Couples the vector ingest to the serializer's output format —
  acceptable because both scripts live in this repo and the test suite would
  catch a format drift.
- **Embed explicitly, hand to Chroma.** Don't let Chroma own an embedding
  function. Keeps the embedding model a single visible swap point —
  `retrievers/vector.py` must embed queries with the *same* model, and naming
  it once in `build_vectors.py` (rather than burying it in collection config)
  makes that contract obvious to whoever writes the retriever next.
- **Defer imports of `httpx`, `chromadb`, `sentence-transformers` into
  `main()`.** The hermetic test suite runs under the `ingest` extra, which
  doesn't (and shouldn't) pull torch. Putting these imports at module top
  would break `pytest --collect-only`. Module-importable for tests; heavy deps
  only on actual execution.
- **`data/chroma/` over `data/embeddings/`.** Considered renaming to a
  role-based path. Rejected: the directory holds a Chroma persistence format
  (sqlite + HNSW index dirs), not raw embeddings; `embeddings/` would mislabel
  the contents. A different backend (FAISS, Qdrant) would write a different
  on-disk format anyway, so the implementation tag is honest. The swap point
  lives in code (`retrievers/vector.py`), not in the directory name.
- **Per-entity file cache as the expensive artifact.** PubMed is rate-limited;
  embedding is cheap. Cache one `.txt` per entity, idempotent re-runs of the
  fetch (skip-existing), full rebuild of the Chroma collection each
  `build_vectors.py` run. Decouples the two scripts cleanly.
