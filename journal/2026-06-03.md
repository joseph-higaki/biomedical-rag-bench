## Decisions (and why)
- **Decision B — full Hetionet graph in GraphDB is canonical for ground truth.**
  A curated slice's answer is true-of-fixture, not true-of-world (`{TSPAN5,
  TTC13}` is not adrenal gland's real expressed set; the real answer is 12,157).
  The producer must derive ground truth from the full graph regardless, so
  per-template validation runs there too — consolidating on one graph rather than
  maintaining two sources of truth. The slice keeps only its step-1 plumbing job.
  Rejected option A (taxonomy-covering hermetic slice): its exact-set assertions
  are arbitrary truncations, and the fixture engineering escalates per question
  type (intersections, differences, reachable paths).
- **Bounded-fan seeds.** Committed example seeds are chosen so the real answer is
  small and enumerable (nasal cavity = 4, Tiludronate = 16) — keeps deterministic
  exact-set validation instead of weak count-only checks. High-fan seeds (adrenal
  gland) would force count+head recording; avoided by seed choice.
- **`type_id` convention.** Zero-padded two-digit prefix for correct lexical sort
  (`10_` after `09_`); `3plus` for type 4's "+" (filename/identifier-safe);
  `type_id` lives as a column in the authoritative taxonomy table so findings and
  templates cite one canonical token.
- **Terminology corrected.** What we're doing per template is *ground-truth
  validation* — a single-seed manual dry-run of the step-3 producer — not "smoke
  testing." "Smoke" now refers only to the step-1 pipeline-plumbing check.
