## Decisions (and why)
- **No version bump for this session.** Per README release strategy, the version axis tracks *result comparability* (MAJOR=breaks it, MINOR=adds capability, PATCH=bug fix that *corrects results*). Nothing here changed a result — numbers are byte-identical (3/9/13/13/16). This is docs + internal analysis-tooling refactor → plain commit to `main`, no tag. Next tag is `v1.1.0` (H4/type-10, appends questions = MINOR).
- **"premium" was a bad name → renamed.** It reads like a price tier when it's an input-token *delta*; that actively undercuts the tokens-vs-dollars hygiene the whole session enforced. Rejected the user's "output tokens" alternative — "output" is the reserved word for generated/output-rate tokens; using it for an input-side quantity mislabels both direction and price tier. Settled on `retrieval_context_input_tokens` (input = right tier, context = the attribution).
- **Safe to rename** despite the telemetry-additive constraint — it's a *derived analysis column* in `load.py`, not a persisted `traversal_info`/result-row key. The constraint governs the eval output contract only; the rename doesn't break result comparability (JSONL untouched).
- **run-cost separate from build-cost, not merged.** Capex (one-time index build, no LLM) vs opex (per-query token spend) are different cost axes; merging would repeat the "single cost scalar" mistake `build-cost.md` already warns against.
- **H5 gets the token leg, but its claim/predicted stay about latency+build.** The token-*efficiency* verdict (where graph is nuanced/partial) belongs to H1; H5 reports the token number descriptively, predicts no winner on it.
- **Price-dim grain:** `(date, model)` with per-tier rate columns, **four** generator tiers (input, cached-read, cache-creation, output) — no `token_type` dimension needed because the fact table is already wide (corrected my own earlier overclaim).
